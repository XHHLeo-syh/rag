## 🚀 各 Demo 详解

### Demo 1: 完整 RAG 基础流程

**目标：** 跑通 RAG 完整链路

**核心流程：**
```
文档读取 → 固定长度分片 → 向量化 → 存储 → 检索 → 生成
```

**技术要点：**
- LangChain4j DocumentLoader 使用
- DocumentSplitter 分片配置
- EmbeddingModel 调用
- InMemoryEmbeddingStore 存储
- 简单相似度检索

**核心原理：**
- RAG 通过检索外部知识注入上下文，减少模型幻觉
- 分片长度影响检索精度与上下文完整性，经验值 200-500 token
- 向量相似度计算：余弦相似度 / 欧氏距离 / 点积

---

### Demo 2: 混合检索实现

**目标：** 向量检索 + 关键词检索融合

**核心流程：**
```
         Query
          │
    ┌─────┴─────┐
    ▼           ▼
向量检索     关键词检索
  Top-10       Top-10
    │           │
    └─────┬─────┘
          ▼
     RRF 融合排序
          │
          ▼
      最终 Top-5
```

**技术要点：**
- BM25 关键词检索实现
- RRF（Reciprocal Rank Fusion）算法
- 加权分数融合

**核心原理：**
- 向量检索：捕获语义相似性，但可能漏掉精确匹配
- 关键词检索：捕获字面匹配，但无法理解语义
- 混合检索：两者互补，提升召回质量

---

### Demo 3: 重排序（Rerank）

**目标：** 两阶段检索提升准确率

**核心流程：**
```
Query → 向量检索 Top-20 → Rerank 模型 → 精排 Top-5 → 生成
         （粗排）          （精排）
```

**技术要点：**
- BGE Reranker 集成
- Cross-Encoder 原理
- 粗排 + 精排性能权衡

**核心原理：**
- Rerank 使用 Cross-Encoder 对 Query 和文档进行联合编码，比 Bi-Encoder 更准确
- 只对 Top-N 做 Rerank 是为了平衡准确率与性能
- Cross-Encoder 计算量大，不适合全量检索

---

### Demo 4: 问题改写 + 查询扩展

**目标：** 优化用户 Query 提升检索效果

**核心流程：**
```
用户原始问题
    │
    ├──→ 口语化转书面化
    │
    └──→ 同义词/相关词扩展
         │
         ▼
    优化后的 Query → 检索
```

**技术要点：**
- LLM 改写 Prompt 设计
- 查询扩展策略
- 效果对比验证

**核心原理：**
- 用户提问往往口语化、模糊，直接检索效果差
- 改写为正式书面语更匹配文档风格
- 同义词扩展可提升召回率，但可能引入噪声

---

### Demo 5: 父子分片（Parent-Child Chunking）

**目标：** 精准检索 + 完整上下文

**核心流程：**
```
原始文档
    │
    ├──→ 大块分片（1000 token）→ 存上下文
    │
    └──→ 小块分片（200 token）→ 建索引
         │
         ▼
    检索小块 → 映射到大块 → 返回完整上下文
```

**技术要点：**
- 父子分片映射关系建立
- 检索后上下文还原
- 元数据关联设计

**核心原理：**
- 小块检索更精准（噪声少）
- 大块提供更完整的上下文（信息多）
- 父子分片结合两者优势

## ⚡ 快速运行

### 环境准备
```bash
JDK 17+
Maven 3.8+
```

### 配置密钥
```yaml
# application.yml
ai:
  llm:
    api-key: YOUR_API_KEY
```

### 运行 Demo
```bash
cd demo-01-basic-rag
mvn spring-boot:run

# 测试接口
curl -X POST http://localhost:8080/api/ask \
  -H "Content-Type: application/json" \
  -d '{"query": "你的问题"}'
```
